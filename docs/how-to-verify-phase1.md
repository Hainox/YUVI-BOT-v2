# Как проверить, что Фаза 1 работает

Эта инструкция — сквозной чеклист для проверки всей Фазы 1 (сбор данных и статистика) на реальном чате. Каждый шаг содержит команду или действие и ожидаемый результат. Если что-то не совпало с ожидаемым результатом — не переходи к следующему шагу, сначала разберись в текущем.

Перед началом убедись, что уже пройдены:
- `docs/botfather-setup.md` — токен бота, права администратора, privacy mode.
- `docs/backfill-setup.md` — нужен только для шага 6 (backfill), можно пройти позже.

## 0. Предусловия

1. Поднята база данных: `docker compose up -d postgres`.
   - **Ожидаемый результат:** `docker compose ps` показывает контейнер `postgres` в статусе `running`/`healthy`.
2. Применены миграции: `docker compose run --rm migrations` (или `alembic upgrade head`, если запускаешь без docker).
   - **Ожидаемый результат:** в логе нет ошибок, последняя строка — что-то вроде `Running upgrade ... -> ..., <имя последней миграции>`.
3. Файл `.env` заполнен (минимум `BOT_TOKEN`, `CHAT_ID`, `DATABASE_URL`).
   - **Ожидаемый результат:** `.env` существует в корне проекта, не пустой.
4. Бот запущен: `docker compose up -d bot` (или `python bot/main.py` локально).
   - **Ожидаемый результат:** в логах бота видно `Start polling` (или аналогичное сообщение об успешном запуске), без исключений.
5. В BotFather отключен privacy mode, бот — администратор тестовой группы (см. `docs/botfather-setup.md`).
   - **Ожидаемый результат:** оба пункта выполнены и проверены отдельно по инструкции botfather-setup.md.

Дальше для проверки таблиц в базе данных подключись через `psql` (или любой GUI-клиент вроде DBeaver/pgAdmin) к `DATABASE_URL` из `.env`.

## 1. Сбор сообщений (DATA-01)

В тестовой группе, где работает бот, по очереди отправь:

1. Обычное текстовое сообщение (например, «привет»).
2. Любую команду (например, `/chatstats`).
3. Пересланное сообщение (форвард) из другого чата или канала.
4. Стикер.

**Проверка в базе данных:**

```sql
SELECT telegram_message_id, content_type, text, is_forwarded, created_at
FROM messages
ORDER BY created_at DESC
LIMIT 10;
```

**Ожидаемый результат:** все 4 отправленных сообщения присутствуют в выдаче, включая команду `/chatstats` — это важно проверить отдельно, потому что именно потеря команд была главной проблемой архитектуры-эталона, которую Фаза 1 должна была исправить (сбор идёт через `outer_middleware`, а не через catch-all-обработчик, который команды бы потерял).

## 2. Реакции (DATA-02)

1. Поставь любую эмодзи-реакцию на одно из сообщений из шага 1.
2. Подожди 5-10 секунд.

**Проверка в базе данных:**

```sql
SELECT r.emoji, r.actor_user_id, m.telegram_message_id, r.updated_at
FROM reactions r
JOIN messages m ON m.id = r.message_id
ORDER BY r.updated_at DESC
LIMIT 5;
```

**Ожидаемый результат:** появилась строка с только что поставленной реакцией.

**Если строка не появилась:** это почти всегда означает, что бот не администратор группы — вернись к разделу «Отдельное требование: права администратора нужны для реакций» в `docs/botfather-setup.md` и проверь права бота ещё раз. Реакции — единственная часть Фазы 1, которая перестаёт работать целиком (не частично) при неверных правах бота.

## 3. Правки сообщений (D-03, append-only)

1. Отправь любое текстовое сообщение.
2. Отредактируй его (измени текст через интерфейс Telegram).

**Проверка в базе данных:**

```sql
-- оригинал должен остаться неизменным
SELECT id, text FROM messages WHERE telegram_message_id = <ID_твоего_сообщения>;

-- правка должна появиться отдельной строкой
SELECT message_id, old_text, new_text, edited_at
FROM message_edits
ORDER BY edited_at DESC
LIMIT 5;
```

**Ожидаемый результат:** `messages.text` содержит ПЕРВОНАЧАЛЬНЫЙ текст (не перезаписан новым), а в `message_edits` появилась новая строка с правкой. Это намеренное архитектурное решение (D-03) — история правок сохраняется полностью, ничего не затирается, что важно для честности статистики.

## 4. Частотные словари слов и эмодзи (DATA-03)

1. Отправь ещё 3-5 сообщений с разными словами и хотя бы одним эмодзи в тексте (например, «привет 😀 как дела сегодня»).

**Проверка в базе данных:**

```sql
SELECT word, count FROM word_frequency ORDER BY count DESC LIMIT 10;
SELECT emoji, count FROM emoji_frequency ORDER BY count DESC LIMIT 10;
```

**Ожидаемый результат:** отправленные слова и эмодзи присутствуют в выдаче со счётчиками, отражающими реальное количество употреблений.

## 5. Статистические команды (STATS-01, STATS-02)

Отправляй по очереди в группе и проверяй, что бот отвечает ПРЯМО В ГРУППЕ (не в личные сообщения — это осознанное решение D-05):

| Команда | Что проверить |
|---|---|
| `/mystats` | Личная статистика отправителя (число сообщений, дни активности) |
| `/chatstats` | Общая статистика чата за всё время |
| `/chatstats 7` | Статистика только за последние 7 дней (проверь, что число меньше или равно `/chatstats` без аргумента) |
| `/who` | Топ участников по активности |
| `/streak` | Текущая серия дней подряд с активностью |
| `/peakday` | День с максимальной активностью чата |

Дополнительно проверь старые команды-алиасы — они должны давать результат, согласованный с новыми:

| Старая команда | Новая (тот же результат) |
|---|---|
| `/stats` | `/mystats` |
| `/top` | `/who` |
| `/activity` | `/peakday` |
| `/words` | топ слов внутри `/chatstats` |

**Ожидаемый результат:** каждая команда отвечает в группе без ошибок; `/stats` даёт тот же результат, что `/mystats` (это один и тот же обработчик под двумя именами, не отдельная реализация — расхождений быть не может).

## 6. Backfill (DATA-04)

Требует настройки `TG_API_ID`/`TG_API_HASH` — см. `docs/backfill-setup.md`, если ещё не сделано.

1. От имени администратора чата отправь `/backfill`.
2. Дождись сообщения бота о запуске и завершении фоновой загрузки.

**Проверка в базе данных:**

```sql
SELECT source, count(*) FROM messages GROUP BY source;
```

**Ожидаемый результат:** появились строки с `source='backfill'` (старые, добытые до того, как бот был в чате) наряду со строками `source='live'` (собранные ботом в реальном времени из шагов 1-4 выше) — и общее число сообщений НЕ задвоилось (ранее собранные вживую сообщения не продублировались backfill'ом благодаря идемпотентной записи по `(chat_id, telegram_message_id)`).

Если запустить `/backfill` ещё раз — дублей быть не должно (проверь тем же запросом: числа не должны увеличиться повторно для уже загруженных сообщений).

## 7. Известные ограничения

Эти пункты — не баги, а осознанные и задокументированные ограничения Фазы 1. Не нужно пытаться «починить» их без явного планирования отдельной фазы.

### Удаление сообщений не детектируется

Telegram Bot API (то, через что работает основной бот) **не присылает никакого события об удалении сообщения** ни в одном типе чатов — это подтверждённое, постоянное ограничение самого Telegram, а не недоработка бота или библиотеки aiogram.

- Колонка `deleted_at` в таблице `messages` присутствует в схеме базы данных (задел на будущее), но в рамках Фазы 1 она всегда остаётся `NULL` — никакого механизма её заполнения нет и не планировалось в этой фазе.
- Если участник чата удаляет своё сообщение, оно продолжает числиться в статистике бота как есть — бот физически не может узнать об удалении через тот способ подключения к Telegram, которым пользуется.
- Реальная детекция удалений технически возможна только через постоянно подключённый MTProto-«слушатель» (на основе того же личного аккаунта, что используется для backfill), который должен работать не разово, а непрерывно, 24/7 — это отдельная, более тяжёлая архитектура, которая может быть рассмотрена в одной из будущих фаз, но не входит в Фазу 1.

### Медиа из backfill нельзя переслать через бота повторно

Как уже описано в `docs/backfill-setup.md`: фото/видео/файлы, загруженные через backfill, помечены `source='backfill'` и сохраняют только `file_unique_id` — их можно учитывать в статистике, но нельзя заново скачать или переслать через самого бота (Bot API и личный MTProto-аккаунт используют разные системы идентификаторов файлов).

## Итог

Если все шаги 1-6 дали ожидаемый результат и раздел 7 понятен — Фаза 1 (сбор данных и статистика) работает полностью и готова быть фундаментом для следующих фаз проекта.
